Architecture

Overview

This project implements a production-inspired autoscaling and resilience platform on Amazon EKS. The objective was not simply to deploy a Kubernetes application, but to understand how modern cloud-native platforms scale, recover, and operate under changing demand.

The platform combines Infrastructure as Code, request-driven autoscaling, dynamic node provisioning, load testing, capacity planning, and failure analysis to study real-world operational scenarios commonly encountered by Site Reliability Engineers and Platform Engineers.

Key goals of the project include:

* Infrastructure provisioning using Terraform
* Dynamic pod scaling using KEDA
* Dynamic node provisioning using Karpenter
* Request-based autoscaling using AWS Application Load Balancer metrics
* Capacity planning and performance validation
* Failure engineering and operational analysis

⸻

Architecture Principles

The platform was designed around the following principles:

Infrastructure as Code

All infrastructure resources are provisioned and managed through Terraform. This ensures repeatability, version control, and consistent deployments.

Demand-Driven Scaling

Scaling decisions are driven by incoming user traffic rather than relying exclusively on CPU or memory utilization. This more closely reflects how production systems experience growth and demand fluctuations.

Dynamic Infrastructure Provisioning

Application scaling and infrastructure scaling are treated as separate concerns. Pods can scale independently, while Karpenter dynamically adjusts node capacity when additional compute resources are required.

Failure-Oriented Learning

A significant portion of the project focuses on understanding how systems fail, how failures manifest, and how engineers can effectively diagnose and recover from those situations.

Core Components

Terraform

Terraform is responsible for provisioning and managing the AWS infrastructure.

Resources managed include:

* VPC
* Subnets
* Route Tables
* NAT Gateway
* EKS Cluster
* IAM Roles and Policies
* IRSA Configuration
* Karpenter Resources

Amazon EKS

Amazon EKS provides the Kubernetes control plane and orchestration capabilities required to run containerized workloads.

The cluster serves as the foundation for application deployment, autoscaling, and resilience testing.

AWS Load Balancer Controller

The AWS Load Balancer Controller integrates Kubernetes with AWS networking services.

Responsibilities include:

* Creating Application Load Balancers
* Managing Target Groups
* Configuring Listener Rules
* Synchronizing Ingress resources with AWS infrastructure

KEDA

KEDA extends Kubernetes autoscaling capabilities by allowing scaling decisions to be based on external metrics.

In this project, KEDA monitors CloudWatch metrics generated by the Application Load Balancer.

Horizontal Pod Autoscaler

The Horizontal Pod Autoscaler adjusts application replica counts based on recommendations generated through KEDA.

This provides workload elasticity as traffic increases or decreases.

Karpenter

Karpenter dynamically provisions worker nodes when additional cluster capacity is required.

Unlike traditional node groups that require predefined scaling configurations, Karpenter reacts directly to scheduling pressure and launches nodes on demand.

Request-Based Autoscaling Strategy

Many Kubernetes environments rely exclusively on CPU utilization for scaling decisions.

While CPU remains an important metric, user traffic often increases before CPU utilization becomes a meaningful indicator of demand.

To address this, the platform uses Application Load Balancer metrics collected through CloudWatch.

The scaling workflow follows this pattern:

Incoming Requests
        |
        v
Application Load Balancer
        |
        v
CloudWatch Metrics
        |
        v
KEDA
        |
        v
HPA
        |
        v
Additional Pods

This approach aligns scaling decisions more closely with actual user demand.

Load Testing

Load testing was performed using k6.

The primary objective was to validate platform behavior rather than identify the absolute throughput limits of the application.

Observed results included:

* Sustained traffic generation
* Successful metric collection
* KEDA-triggered scaling events
* HPA-based pod scaling
* Dynamic node provisioning

The platform successfully handled approximately 1,125 requests per second during testing.

Failure Engineering Focus Areas

Several production-inspired failure scenarios were explored as part of the project.

HTTP 503 Errors

Studied situations where no healthy backend targets were available behind the load balancer.

Connection Refused

Analyzed TCP-level failures where a destination host is reachable but no application is listening on the target port.

Connection Timeout

Investigated network-level failures involving dropped packets, routing issues, and delayed responses.

Spot Interruptions

Reviewed interruption notices, node draining workflows, and workload recovery behavior in Spot environments.

IP Exhaustion

Explored Amazon EKS networking limits including:

* ENI limits
* Pod density constraints
* Prefix Delegation
* Subnet capacity
* VPC address planning

Key Takeaways

This project demonstrates that modern autoscaling platforms involve far more than simply increasing replica counts.

Successful scaling depends on coordination between multiple systems, including:

* CloudWatch
* KEDA
* HPA
* Kubernetes Scheduler
* Karpenter
* EC2
* VPC Networking
* Application Load Balancers

Understanding how these components interact under both normal and failure conditions is critical to designing reliable and cost-effective cloud-native platforms.
