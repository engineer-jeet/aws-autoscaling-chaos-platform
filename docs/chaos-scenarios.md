Chaos Scenarios

Overview

One of the primary goals of this project was to move beyond successful deployments and understand how cloud-native platforms behave during failure conditions.

In production environments, outages are rarely caused by a single component. Failures often emerge from the interaction between networking, infrastructure, load balancers, applications, and orchestration systems.

The objective of these exercises was not to intentionally break the platform, but to understand how common failure scenarios manifest, how they can be identified, and how engineers should respond during incident situations.

Each scenario was selected because it represents a failure pattern frequently encountered in modern Kubernetes environments.

Scenario 1 - HTTP 503 Service Unavailable

Objective

Understand how AWS Application Load Balancer behaves when no healthy backend targets are available.

Experiment

The application deployment was scaled to zero replicas.

As a result:

* No application pods remained available.
* Target groups contained no healthy targets.
* The Application Load Balancer continued receiving requests.

Observed Behavior

Users received:

HTTP 503 Service Unavailable

The response was generated by the load balancer rather than the application itself.

Root Cause

The load balancer was functioning correctly but had no healthy destinations to route traffic to.

Request path:

Client
    ↓
ALB
    ↓
No Healthy Targets
    ↓
HTTP 503

Production Examples

This scenario commonly occurs during:

* Failed deployments
* Readiness probe failures
* Pod crashes
* Aggressive scale-in events
* Availability Zone outages

Key Learning

A 503 error does not necessarily indicate an application defect.

In many cases, it indicates a capacity or availability problem somewhere in the request path.

Scenario 2 - Connection Refused

Objective

Understand TCP-level failures where the destination host is reachable but the target service is unavailable.

Observed Behavior

The client successfully reached the destination host.

However:

* No process was listening on the requested port.
* The operating system immediately rejected the connection.

Result:

Connection Refused

TCP Behavior

The server responds with a TCP Reset (RST).

Request path:

Client
    ↓
Host Reachable
    ↓
Port Closed
    ↓
TCP RST
    ↓
Connection Refused

Production Examples

This commonly occurs when:

* Applications crash
* Containers fail to start
* Services bind to incorrect ports
* Processes terminate unexpectedly

Key Learning

Connection Refused is generally an application or service availability problem rather than a networking problem.

The request successfully reached the host.

The failure occurred after network connectivity had already been established.

Scenario 3 - Connection Timeout

Objective

Understand failures caused by network path disruption or packet loss.

Observed Behavior

The client attempted to establish a connection.

However:

* Packets were dropped
* Responses never returned
* Connection establishment never completed

Result:

Connection Timed Out

Request Path

Client
    ↓
Network Path
    ↓
Packet Drop
    ↓
No Response
    ↓
Timeout

Production Examples

Connection timeouts commonly occur because of:

* Security Group restrictions
* Network ACL rules
* Routing issues
* Firewall policies
* Network congestion
* Infrastructure failures

Key Learning

Connection Timeout usually indicates a networking problem rather than an application problem.

Unlike Connection Refused, the request never successfully reaches the destination service.

Scenario 4 - Spot Interruption Analysis

Objective

Understand how Kubernetes workloads behave when AWS Spot capacity becomes unavailable.

Topics Studied

* Spot Interruption Notices
* Rebalance Recommendations
* Node Draining
* Pod Eviction
* Workload Recovery

Typical Lifecycle

Spot Instance
        ↓
Interruption Notice
        ↓
Node Cordoned
        ↓
Pods Evicted
        ↓
Replacement Capacity Provisioned
        ↓
Workload Recovery

Production Impact

Without interruption handling:

* Workloads terminate abruptly
* User requests fail
* Capacity drops unexpectedly

With proper handling:

* Pods are drained gracefully
* Replacement nodes are provisioned
* User impact is minimized

Key Learning

Spot capacity should be treated as interruptible by design.

Production workloads require interruption-aware architecture rather than assuming Spot instances will remain available indefinitely.

Scenario 5 - IP Exhaustion

Objective

Understand networking limits within Amazon EKS.

Topics Studied

* Elastic Network Interfaces (ENIs)
* Pod Density Limits
* Prefix Delegation
* Subnet Capacity
* VPC Address Planning

Failure Pattern

A cluster may appear healthy from a compute perspective while still being unable to schedule new workloads.

Example:

CPU Available
Memory Available
Node Available

But:

No Available Pod IP Addresses

Result:

Pods Remain Pending

Production Symptoms

Common indicators include:

* Pods stuck in Pending state
* Failed pod scheduling
* AWS CNI allocation errors
* IP allocation failures
* Karpenter scaling without successful scheduling

Why This Matters

Networking limits are often overlooked during capacity planning.

Many engineers focus on:

* CPU
* Memory
* Node counts

while ignoring:

* ENI limits
* Pod IP availability
* Subnet exhaustion

In large EKS environments, networking limits frequently become the first scaling bottleneck.

Key Learning

Successful scaling requires more than compute capacity.

Networking resources must be planned and monitored with the same level of attention as CPU and memory.

Cross-Scenario Observations

Several themes emerged across all scenarios.

User Symptoms Rarely Reveal Root Cause

A user may only observe:

Application Unavailable

while the actual cause may be:

* Missing pods
* Failed nodes
* Network restrictions
* IP exhaustion
* Load balancer health failures

Infrastructure Problems Often Appear As Application Problems

Many failures initially look like application defects even when the underlying issue exists elsewhere in the stack.

Observability Is Critical

Without metrics, logs, and monitoring data, identifying root causes becomes significantly more difficult.

Fast diagnosis requires visibility into:

* Infrastructure
* Networking
* Kubernetes
* Applications
* Cloud services

Failure Analysis Accelerates Learning

Successful deployments teach engineers how systems work.

Failures teach engineers how systems behave under stress.

Both perspectives are necessary to build reliable platforms.

⸻

Key Takeaways

The exercises performed during this project reinforced several important principles:

1. Reliability engineering requires understanding failure behavior, not just successful operation.
2. Modern outages often involve multiple layers of the technology stack.
3. Application availability depends on infrastructure, networking, orchestration, and cloud services working together.
4. Capacity planning and resilience engineering are closely related disciplines.
5. Understanding why systems fail is often more valuable than understanding why they succeed.

These scenarios provided practical insight into the operational challenges associated with running scalable Kubernetes workloads in production environments.
